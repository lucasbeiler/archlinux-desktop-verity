## Details
Verified Boot is implemented using UEFI Secure Boot, TPM 2.0, and dm-verity. Kernel and initramfs are combined according to the Unified Kernel Image (UKI) specification. Alongside the bootloader, this unified image is cryptographically validated by UEFI Secure Boot during every boot cycle, ensuring the integrity and authenticity of the early boot chain. The immutable, read-only root partition is cryptographically validated by dm-verity at each boot, providing robust integrity and authenticity guarantees for the entire root partition. Any modification to the bootloader, the Unified Kernel Image, or the root partition will disrupt the boot process, thereby enforcing a strict chain of trust. TPM 2.0 acts as a hardware root of trust, ensuring that user data decryption keys are only accessible if the firmware binaries and Secure Boot configurations match known-good, PCR-based measurements. The user data partition (`/data`) is the only writable and persistent area on the disk; in the future, IPE (Integrity Policy Enforcement) policies will strictly prevent execution from it. As a result, the system's entire executable surface is cryptographically validated.

Regarding application security, aside from a Wayland compositor/window manager and a few basic utilities, Chromium is the primary user-facing application. It features, by default, a robust sandboxing architecture and advanced, modern exploit mitigations (see the [Rule of 2](https://chromium.googlesource.com/chromium/src/+/master/docs/security/rule-of-2.md)). For example, Chromium uses [PartitionAlloc](https://chromium.googlesource.com/chromium/src/+/master/base/allocator/partition_allocator/PartitionAlloc.md) as its secure memory allocator and deploys modern exploit mitigations such as Control Flow Integrity (CFI), shadow stacks, and memory tagging (on supported hardware). Beyond the browser's own robust sandbox, the OS further restricts Chromium using AppArmor. Similarly, system services with higher attack surfaces, such as `wpa_supplicant`, are confined via AppArmor and systemd's built-in sandboxing features. Chromium is hardened even further by [running in JIT-less mode](https://microsoftedge.github.io/edgevr/posts/Super-Duper-Secure-Mode/#:~:text=roughly%2045%25%20of%20CVEs%20issued%20for%20V8%20were%20related%20to%20the%20JIT%20engine). Looking ahead, Chromium's WebAssembly capabilities (even when running [jitlessly](https://docs.google.com/document/d/1OIJ4Sv2XfTlI5NmTS1QI8v8wPL0LUT5s1W2D9OlJmMc/)) could position it as a highly secure, sandboxed runtime for C, C++, Rust, and Go software compiled to WebAssembly, a growing trend.

The kernel incorporates hardening patches from the [linux-hardened](https://github.com/anthraxx/linux-hardened/commits/6.18) project, further hardened by sysctl flags and kernel boot parameters. Security recommendations from the Kernel Self-Protection Project (KSPP), GrapheneOS, PaX/grsec, and the `kernel-hardening-checker` are also applied wherever possible.

The USB and peripheral attack surface is minimized by configuring the kernel's USB subsystem to deny new USB device connections by default (a sysctl toggle provided by [linux-hardened](https://github.com/anthraxx/linux-hardened/commit/7a8596bdd792a7f942ab2ee8f332fa80efc2e7dd)), effectively mitigating against malicious, USB-based exploits. A minimal SUID binary can be invoked to temporarily lift this restriction via sysctl for a few seconds when a user needs to connect a device. Additionally, IOMMU enforcement is enabled to protect against DMA attacks, and known DMA-vulnerable kernel modules (such as Thunderbolt) have been blacklisted. Finally, an automated mechanism reboots the computer if it remains locked for too long, placing the user data partition back at rest in its encrypted state.

A stateful firewall is configured by default using iptables. It permits only outbound connections and drops all unrequested incoming packets. This ensures that only locally initiated traffic can establish network connections, substantially reducing exposure to remote attacks. To further minimize the network attack surface, IPv6 is entirely disabled in the kernel. Because IPv6 has not yet fully replaced IPv4, restricting the network stack exclusively to IPv4 reduces the attack surface.

Secure DNS is configured system-wide using [dnscrypt-proxy](https://github.com/DNSCrypt/dnscrypt-proxy). Furthermore, Chromium's own DNS resolver is configured to use DNS over HTTPS (DoH) via the AdGuard DNS endpoint, which blocks advertisements and malicious domains. Chromium also utilizes ECH/ESNI to prevent domain name leakage through unencrypted SNI. For secure network time synchronization, the system uses [ntpd-rs](https://github.com/pendulum-project/ntpd-rs), an NTP/NTS client written in Rust. It leverages the Network Time Security (NTS) protocol rather than standard NTP to ensure authenticated and secure time syncing.

Arch Linux serves as an excellent foundational distribution, featuring a high percentage of packages that pass reproducible build tests. This provides verifiable transparency, ensuring that distributed binaries precisely match the source code. Additionally, Arch maintainers adhere to excellent security practices by compiling packages with robust compiler-enforced mitigations and keeping software closely aligned with upstream releases. This project leverages those benefits to achieve a Verified Boot implementation on Arch Linux.
